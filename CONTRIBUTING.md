# Contributing

Thanks for considering a contribution. The most useful contributions are bug
reports with reproductions, performance improvements with benchmarks, and
additional test vectors. Feature work — especially anything touching the spec
or wire format — should start as an issue so design can be discussed before
code is written.

## Ground rules

The wire format is the contract. Any change that produces a different
canonical byte sequence for an existing envelope is a breaking change and
needs a `schemaVersion` bump. The CI enforces this: if regenerating the
committed test vectors changes any byte, the build fails.

This SDK has a sibling: the Python SDK at
[sigill-python](https://github.com/sigill-ai/sigill-python). Both repos vendor
the same `spec/` directory, byte-for-byte. If you change canonicalization,
hashing, or the JSON Schema here, you must open a coordinated PR in
sigill-python too — otherwise envelopes sealed by one SDK will fail to
verify with the other, and that's the worst kind of bug for a project like
this.

## Local development

```
dotnet test
```

Targets `net8.0`, `net9.0`, and `netstandard2.1`. CI runs on Ubuntu and
Windows.

## Updating the spec

The JSON Schema lives at `spec/ai-evidence-envelope-v1.schema.json`. Test
vectors are generated by the `_generate.py` scripts in each
`spec/test-vectors/*/` directory; they require Python 3.9+ and the
`jcs` package (`pip install jcs`). To regenerate after a schema change:

```
python spec/test-vectors/01-complete-ai-call/_generate.py
python spec/test-vectors/02-pii-redacted/_generate.py
```

Then copy the entire `spec/` directory verbatim into sigill-python in the
same coordinated PR.

## Pull request checklist

- [ ] Tests added for the new behavior.
- [ ] `dotnet test` passes locally on net8.0 and net9.0.
- [ ] If you changed the spec or canonicalization, you regenerated the test
      vectors and opened a matching PR in sigill-python.
- [ ] If you added a public API, you also added XML doc comments and updated
      `README.md`.
- [ ] Commit messages explain *why*, not just *what*.

## Reporting security issues

Please do not file public issues for security vulnerabilities. Email
`security@sigill.ai` with details and reproduction.
